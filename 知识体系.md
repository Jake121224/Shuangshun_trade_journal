```
你（用户）
  │
  ▼
手机浏览器 / 电脑浏览器
  │

zumingtradelog.netlify.app  ←── Netlify（存 HTML 文件）
```

## 第一章：浏览器是怎么工作的

### 1.1 三个基础语言

想象你在搭积木：

| 语言 | 角色 | 类比 |
|------|------|------|
| **HTML** | 骨架 | 房子的墙壁、门窗、楼层 |
| **CSS** | 装修 | 墙壁颜色、窗帘样式、灯光 |
| **JavaScript** | 电器 | 开关、遥控器、自动门 |

你的 `trade-journal.html` 里三个都有，所以能看、能点、能存数据。

### 1.2 浏览器怎么拿到这个文件

```
你在地址栏输入 zumingtradelog.netlify.app
           │
           ▼
      DNS（电话簿）：把域名翻译成 IP 地址
           │
           ▼
      Netlify 服务器：找到 index.html，发回给你
           │
           ▼
      你的浏览器：从第一行读到最后一，行画出页面
```

**关键概念**：
- **服务器**：一台永远不关机的电脑，存着 HTML 文件
- **DNS**：把 `zumingtradelog.netlify.app` 翻译成 `192.0.2.1` 这种数字地址
- **HTTP**：浏览器和服务器之间的通讯语言

### 1.3 你的页面是"活"的

普通 HTML 只能展示死内容。加了 JavaScript 才能：

- 点击按钮切换 Tab（录入/历史/统计）
- 读取输入框里的数据
- 把数据存到 localStorage 或发给 Supabase
- 画 SVG 图表

## 第二章：数据存哪里？localStorage vs 云端

### 2.1 localStorage —— 浏览器自带的抽屉

```
Chrome 浏览器
  └── localStorage
        └── "trade_journal_v1" → [{交易1}, {交易2}, {交易3}]
```

**特点**：
- ✅ 不需要网络
- ✅ 读写速度快
- ❌ 只在这一个浏览器的一个设备上有
- ❌ 清浏览器缓存就没了
- ❌ 换手机看不到

### 2.2 Supabase —— 云端的数据库

```
你的手机 ──→ internet ──→ Supabase 服务器
                                │
                                └── PostgreSQL 数据库
                                       │
                                       └── trades 表
                                             ├── id: "t123456"
                                             ├── trade: { 交易数据 }
                                             └── created_at: "2026-07-03"
```

**特点**：
- ✅ 所有设备都能访问
- ✅ 数据永久保存
- ❌ 需要网络
- ❌ 比 localStorage 慢一点

### 2.3 你的项目用了"双保险"

```
打开页面
  │
  ├─ 1) 先从 localStorage 读 ──→ 页面秒开，用户看到数据
  │
  └─ 2) 再从 Supabase 读 ──→ 拿到最新数据，覆盖本地
       │
       └─ 如果 Supabase 读失败 ──→ 用本地的，不影响使用
```

**这就是离线也能用的秘密**。

## 第三章：Supabase 是什么

### 3.1 一句话

Supabase = **免费的后端**。你不用自己写服务器代码，它给你提供：

| 服务 | 你的项目用在哪 |
|------|--------------|
| **数据库** (PostgreSQL) | 存交易记录 (`trades` 表) |
| **API** (自动生成) | 增删改查数据，不需要写后端代码 |
| **存储** (Storage) | 存截图文件 |

### 3.2 数据库怎么存数据

想象一张 Excel 表格：

```
┌──────────────┬─────────────────────┬─────────────────────┐
│ id (主键)     │ trade (JSONB)        │ created_at           │
├──────────────┼─────────────────────┼─────────────────────┤
│ t1765432109  │ {symbol:"螺纹 RB2510"│ 2026-07-03 14:30    │
│              │  dir:"多",...}       │                     │
│ t1765432110  │ {symbol:"沪银 ag"...│ 2026-07-03 15:00    │
│              │ }                   │                     │
└──────────────┴─────────────────────┴─────────────────────┘
```

- **id**：每笔交易的身份证号
- **trade**：一个 JSONB 列（可以存整个对象，不需要拆分字段）
- **created_at**：时间戳

### 3.3 怎么跟数据库说话 —— API

```
JavaScript 代码:
  sb.from('trades').select('*')     → 翻译成 SQL: SELECT * FROM trades
  sb.from('trades').upsert({...})   → 翻译成 SQL: INSERT ... ON CONFLICT UPDATE
  sb.from('trades').delete()...     → 翻译成 SQL: DELETE FROM trades WHERE ...
```

Supabase 自动把你的 JavaScript 翻译成数据库能懂的 SQL。这就是"后端即服务"。

### 3.4 Key 是什么

```
SB_URL  = 'https://tawovxvjcywlkfawoshg.supabase.co'  ← 数据库地址
SB_KEY  = 'sb_publishable_...'                          ← 访客通行证（公开）
```

- `publishable` key 是**公开的**，放在 HTML 里没关系（它只能操作你开了权限的表）
- `service_role` key 是**秘密的**，相当于管理员密码，永远不要放前端代码里

### 3.5 RLS（行级安全策略）是什么

如果不设置 RLS，Supabase 默认拒绝所有请求。你跑的这两条 SQL 就是在开门：

```sql
-- 允许所有人操作 trades 表
CREATE POLICY "allow_all" ON trades FOR ALL USING (true);

-- 允许所有人操作 screenshots 存储桶
CREATE POLICY "public_screenshots" ON storage.objects FOR ALL
  USING (bucket_id = 'screenshots')
  WITH CHECK (bucket_id = 'screenshots');
```

因为是个人项目，开了全部权限。如果是多人用的 App，就需要限制每个人只能看到自己的数据。

## 第四章：Netlify 是怎么托管页面的

### 4.1 什么是"托管"

你的 `trade-journal.html` 在电脑上：
```
/Users/jake/Documents/futures/trade-journal.html
→ 只有你能打开（在你的电脑上）
```

上传到 Netlify 后：
```
https://zumingtradelog.netlify.app
→ 全世界任何设备都能打开
```

Netlify 做了一件事：**把你的文件放到一台公共服务器上**。

### 4.2 CDN —— 为什么打开这么快

```
你的 HTML 文件被复制到全球几十个节点：

旧金山 ─┐
伦敦   ─┤
东京   ─┼── 都有你的 index.html 副本
新加坡 ─┤
悉尼   ─┘

用户从最近的节点获取 → 更快
```

这就是 **CDN（内容分发网络）**。Netlify 自动帮你做了。

### 4.3 HTTPS —— 为什么地址栏有锁

```
HTTP   → 明文传输，像明信片，沿途谁都能看
HTTPS  → 加密传输，像密封信封，只有收发双方能看
```

Netlify 自动提供免费 HTTPS 证书。你的 Supabase 通讯也是 HTTPS 加密的。

## 第五章：Git & GitHub —— 代码的时光机

### 5.1 Git 是什么

```
每次 commit = 给代码拍一张快照

commit 377da57: "feat: 初始化交易复盘工具"
    ↓
commit 18eb84e: "chore: 添加 .gitignore"
    ↓
commit 72a59f9: "feat: 添加截图功能 + Supabase 云同步"
    ↓
commit f4d2b65: "fix: 云端空时自动上传本地数据"
```

你可以随时：
- 回到任何一个快照（`git checkout 377da57`）
- 看每次改了什么（`git diff`）
- 对比两个版本的差异

### 5.2 GitHub 是什么

```
Git = 你电脑上的工具（本地）
GitHub = 微软的网站（远程），用来：
          - 备份你的 Git 仓库
          - 分享给别人看
          - 多人协作
```

`git push` = 把本地的提交推到 GitHub

### 5.3 .gitignore 是什么

有些事情不需要记录：
```
.DS_Store     ← Mac 系统文件
*.log         ← 日志文件
```

`.gitignore` 告诉 Git："这些文件别管它们"。

## 第六章：你的应用是怎么跑起来的

### 6.1 完整生命周期

```
┌─────────────────────────────────────────────────────┐
│                   打开页面                            │
├─────────────────────────────────────────────────────┤
│ 1. HTML/CSS 渲染 → 看到界面                          │
│ 2. Supabase SDK 加载                                 │
│ 3. 从 localStorage 秒读缓存 → 立刻显示数据            │
│ 4. 从 Supabase 拉最新数据 → 覆盖缓存                  │
│ 5. 如果本地有数据但云端是空的 → 自动上传到云端         │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│                   录入一笔交易                         │
├─────────────────────────────────────────────────────┤
│ 1. 填表单：品种、价格、方向、止损...                  │
│ 2. 仓位计算器：资金 × 风险% ÷ 止损点数 = 建议手数     │
│ 3. 贴截图：Ctrl+V → Blob → 暂存内存                  │
│ 4. 点保存                                            │
│    ├─ 构造 trade 对象                                │
│    ├─ 写内存数组 trades[]                             │
│    ├─ 写 localStorage 缓存                           │
│    ├─ 写 Supabase trades 表                          │
│    └─ 上传截图 → Supabase Storage → 拿回 URL → 更新记录│
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│                   换设备打开                           │
├─────────────────────────────────────────────────────┤
│ 1. 新设备 localStorage 是空的                        │
│ 2. 从 Supabase 拉到全部数据                          │
│ 3. 写入 localStorage 做缓存                          │
│ 4. 和原来的设备看到一样的数据                         │
└─────────────────────────────────────────────────────┘
```

### 6.2 盈亏 R 是怎么算的

```
R = 盈亏点数 ÷ 止损点数

例子（做多）：
  进场价: 3287     止损价: 3268     止损 = 19 点
  出场价: 3325     盈亏 = 3325 - 3287 = +38 点
  R = 38 ÷ 19 ≈ +2.0R

做空反过来：
  进场 3287，出场 3250
  盈亏 = 3287 - 3250 = +37 点
  R 同样按止损点数算
```

**R 的好处**：不看赚了多少元，只看风险回报比。1R = 你愿意亏的金额。+2R = 赚了两倍风险金。

### 6.3 截图怎么上传的

```
1. 你按 Ctrl+V
      │
2. 浏览器拿到粘贴的图片 → Blob（二进制大对象）
      │
3. 存在 pendingShots[] 数组里 → 页面显示缩略图预览
      │
4. 你点保存
      │
5. 调用 uploadShots(tradeId)
      │
6. sb.storage.from('screenshots').upload(path, file)
      │
7. 拿到 publicUrl → 存进 trade.screenshots
      │
8. 再次更新 Supabase 记录
```

## 第七章：为什么这样设计 —— 架构决策

### 7.1 单文件 HTML 而不是 React/Vue

| | 单文件 HTML | React/Vue 框架 |
|---|---|---|
| 部署 | 拖到 Netlify 即可 | 需要构建工具 |
| 修改 | 记事本打开就能改 | 需要 Node.js 环境 |
| 理解 | 一个文件看完 | 几十个文件 |
| 适合 | 个人小工具 | 大型应用 |

**你的场景**：个人用、功能集中、快速迭代 → 单文件最合适。

### 7.2 Supabase 而不是自建后端

| | Supabase | 自建后端 |
|---|---|---|
| 费用 | 免费 500MB | 服务器月费 ¥50+ |
| 维护 | 零 | 需要运维 |
| API | 自动生成 | 手写 |

### 7.3 localStorage 双写的意义

这是一个 **"离线优先"** 的策略：

```
网络断了 → localStorage 还能用
网络恢复 → 自动同步到 Supabase
```

专业术语叫 **Optimistic Local + Server Sync**。

### 8.3 核心概念速查表

| 概念 | 一句话解释 |
|------|-----------|
| **前端** | 浏览器里跑的（HTML/CSS/JS） |
| **后端** | 服务器上跑的（Supabase 帮你做了） |
| **数据库** | 存数据的地方（PostgreSQL） |
| **API** | 前后端之间的"接线口" |

| **SVG** | 用代码画图（你的图表就是用 SVG 画的） |
| **Git** | 代码的版本管理 |
| **Commit** | 给代码拍一张快照 |

## 总结

你不需要一次性全懂。每次改代码时，翻回来看对应章节就行。

这张地图的核心是三条线：

```
界面线：HTML → CSS → 浏览器渲染 → 用户看到
数据线：输入 → localStorage 缓存 → Supabase 云存 → 多设备同步
代码线：写代码 → Git commit → GitHub 备份 → Netlify 上线
```

三条线打通，你就能从"会用"升级到"会改"再到"会造"。
